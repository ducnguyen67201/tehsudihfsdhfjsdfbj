## Problem Statement

When a customer sends multiple standalone Slack messages in rapid succession ("hey", "our API is down", "getting 500 errors"), TrustLoop creates a separate conversation for each message. Threaded replies group correctly via `thread_ts`, but standalone messages each get their own `messageTs` as the thread anchor, producing 3 conversations instead of 1. This clutters the on-call engineer's kanban board and fragments context.

## Demand Evidence

- Founder's own company experiences this daily — customers send rapid-fire standalone messages in `#ext-customer-*` channels
- Every competitor (Pylon, Thena, ClearFeed, Plain) has solved this problem, confirming it's table-stakes for Slack-based support tools
- Thena (2min window) and ClearFeed (5min window, AI layer) have validated the time-window approach in production
- Without grouping, the on-call engineer sees 3-5 conversations that are actually one issue, diluting the value of thread risk scoring

## Status Quo

Current code in `apps/queue/src/domains/support/support.activity.ts:13-20`:
- `buildCanonicalConversationKey()` uses `{installationId}:{teamId}:{channelId}:{threadTs}`
- For standalone messages, `threadTs` falls back to `messageTs` (event-normalizer.ts:52)
- Each standalone message = unique conversation key = new conversation via upsert
- Unique constraint on `(workspaceId, canonicalConversationKey)` enforces this

## Target User & Narrowest Wedge

Same as prior design: the on-call engineer at a 10-100 person company handling customer Slack channels. This feature reduces noise in their kanban view by grouping related standalone messages into a single conversation.

## Constraints

- Solo technical founder, pre-launch
- Must not break existing threaded message grouping (thread_ts-based)
- Must not introduce AI dependencies (Phase 1 is deterministic only)
- Must be compatible with future Phase 2 AI grouping and Phase 3 thread nudging
- Temporal workflow must remain deterministic (grouping logic in activities, not workflows)

## Premises

1. Time-window grouping (Thena-style) is the right Phase 1 approach — validated by competitors, deterministic, transparent.
2. Grouping is per-author — industry standard for deterministic grouping (Thena, ClearFeed rule-based). Cross-author incident grouping is a Phase 2 AI feature.
3. Conversations with status UNREAD, IN_PROGRESS, or STALE are eligible for grouping. DONE conversations are never reopened by grouping logic.
4. First message's `messageTs` becomes the `threadTs` anchor for grouped standalone messages. Accepted risk: if someone threads on message #2 or #3, Slack's `thread_ts` won't match the grouped key. Mitigation: manual merge in Phase 2.
5. Time window is configurable per `SupportInstallation` via `metadata.groupingWindowMinutes` (default: 5 minutes).

## Approaches Considered

### Approach A: Query-Before-Insert (Minimal)
Before creating a standalone conversation, query for a recent UNREAD/IN_PROGRESS conversation from the same author in the same channel within the time window. If found, reuse its key. Smallest diff (only changes support.activity.ts), no migration needed. Risk: race conditions with simultaneous messages, no audit trail.

### Approach B: Grouping Anchor Table (CHOSEN)
Add a `SupportGroupingAnchor` table that maps `(channelId, authorSlackUserId, windowStart)` → `conversationId`. On each standalone message, look up the anchor. If found and within window, use that conversation. If not, create new anchor + conversation. Eliminates race conditions, provides clean audit trail, scales to Phase 2 AI grouping.

### Approach C: Create-then-Merge
Keep current flow, create one conversation per message, then immediately merge with a recent sibling. Builds merge logic needed for Phase 2 but is wasteful (2 DB writes instead of 1) and has a brief window where duplicate conversations appear in UI.

## Recommended Approach

**Approach B: Grouping Anchor Table.**

### Schema Changes

New table `SupportGroupingAnchor`:

```prisma
model SupportGroupingAnchor {
  id                String   @id @default(cuid())
  workspaceId       String
  installationId    String
  channelId         String
  authorSlackUserId String
  conversationId    String
  anchorMessageTs   String   // First message's messageTs (becomes the threadTs anchor)
  windowStartAt     DateTime // When this grouping window opened
  windowExpiresAt   DateTime // windowStartAt + groupingWindowMinutes
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  conversation SupportConversation @relation(fields: [conversationId], references: [id])

  // anchorMessageTs in the unique key prevents duplicate anchors for the same
  // first message. The lookup query uses the covering index below, not this constraint.
  @@unique([workspaceId, channelId, authorSlackUserId, anchorMessageTs])
  @@index([workspaceId, channelId, authorSlackUserId, windowExpiresAt])
  @@index([conversationId])
}
```

Add to `SupportInstallation.metadata`:
```json
{
  "groupingWindowMinutes": 5
}
```

### Activity Changes

In `runSupportPipeline()` (support.activity.ts):

1. After normalizing the Slack event, check if the message is standalone (threadTs === messageTs).
2. If standalone:
   a. If `normalized.slackUserId` is null, skip grouping — fall through to current behavior (create new conversation). The anchor requires a non-null author.
   b. Query `SupportGroupingAnchor` for the same `(workspaceId, channelId, authorSlackUserId)` where `windowExpiresAt > now()` and the linked conversation status is UNREAD, IN_PROGRESS, or STALE.
   c. If anchor found: use the anchor's `anchorMessageTs` as the `threadTs` for the canonical key. Extend `windowExpiresAt` using: `windowExpiresAt = max(windowExpiresAt, now() + groupingWindowMinutes)`. Cap at `windowMaxDurationMinutes` (default: 60) from `windowStartAt`.
   d. If no anchor: proceed with current behavior (create new conversation), then create a new anchor pointing to it.
   e. **Transaction boundary:** The anchor lookup (`SELECT ... FOR UPDATE`), anchor upsert, and conversation upsert must all occur within a single `prisma.$transaction()` to prevent TOCTOU races between concurrent messages.
3. If threaded (threadTs !== messageTs): proceed with current behavior unchanged.

**Field mapping:** `authorSlackUserId` in the anchor maps to `normalized.slackUserId` from the event normalizer output.

### Sliding vs Fixed Window

Use a **sliding window with a max cap**: each new message extends `windowExpiresAt` using `max(windowExpiresAt, now() + groupingWindowMinutes)`. This handles the case where a customer sends messages every 4 minutes for 20 minutes — they all belong to one conversation.

**Max cap:** `windowMaxDurationMinutes` (default: 60 minutes, configurable per installation). If `now() - windowStartAt > windowMaxDurationMinutes`, the window expires regardless of sliding. This prevents unbounded conversations from accumulating across hours or days.

### Configuration

Read `groupingWindowMinutes` from `SupportInstallation.metadata` at pipeline start. Default to 5 minutes if not set.

### Edge Cases

1. **Thread reply to grouped message:** If someone threads on message #2 of a group, Slack sends `thread_ts` = message #2's ts. This creates a separate conversation from the grouped one. Accepted for MVP. Mitigation: Phase 2 manual merge.
2. **Non-customer messages:** Only group messages where `normalized.authorRoleBucket` is `SUPPORT_AUTHOR_ROLE_BUCKET.customer`. Skip grouping for `bot`, `system`, and `internal` roles — these are either automated messages or internal team messages that shouldn't be grouped with customer conversations.
3. **Null slackUserId:** If the normalized event has a null `slackUserId`, skip grouping entirely and fall through to the current behavior (create a new conversation). The anchor table requires a non-null author.
4. **DONE conversations:** Never reopen. If the anchor's conversation is DONE, create a new conversation and new anchor.
5. **Anchor cleanup:** Expired anchors are excluded by the `windowExpiresAt > now()` filter in the lookup query. No eager cleanup needed. Tech debt note: add a periodic sweep or TTL-based deletion when the table exceeds ~100K rows (expected well after Phase 2).
6. **Config changes with active anchors:** When `groupingWindowMinutes` is changed on an installation, existing anchors honor their originally-computed `windowExpiresAt`. Only new anchors use the updated config. No retroactive recalculation needed.
